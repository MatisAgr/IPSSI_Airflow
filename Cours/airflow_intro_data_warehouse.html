Objectif pédagogique

Pourquoi ce module avant Airflow ?

Problème fréquent

  • Les étudiants savent lancer un DAG, mais ne savent pas toujours pourquoi la donnée doit être découpée en couches.
  • Ils confondent souvent ingestion, transformation, stockage, analytique et visualisation.
  • Ils produisent des pipelines qui “marchent”, mais qui ne sont ni auditables, ni rejouables, ni utiles au métier.

But de l’introduction

  • Donner le vocabulaire décisionnel de base.
  • Expliquer la logique d’un Data Warehouse et d’un Lakehouse simple.
  • Relier chaque notion à une bonne pratique Airflow.
  • Préparer les TP : API → bronze → silver → gold → analytics → dashboard ou alerte.
Airflow ne donne pas de sens à la donnée. Il orchestre une chaîne qui doit déjà avoir une logique métier, technique et analytique.

BI

Business Intelligence : définition utile

Ce que c’est

  • Ensemble de méthodes, modèles et outils pour transformer des données en informations utiles au pilotage.
  • Elle sert à suivre des indicateurs, analyser des tendances, comparer des performances et détecter des anomalies.
  • Elle n’est pas seulement un outil de visualisation.

Ce que ça produit

  • KPI : mesure clé orientée décision.
  • Dashboard : interface synthétique de pilotage.
  • Rapport : analyse plus détaillée, souvent périodique.
  • Table analytique : structure préparée pour répondre vite à des questions métier.
La BI ne commence pas dans Power BI ou Metabase. Elle commence dans la façon dont les données sont préparées.

Un pipeline analytique ne doit pas simplement recopier l’opérationnel. Il doit produire une donnée adaptée à la décision.
OLTP / OLAP

Deux mondes à ne pas confondre

OLTP — transactionnel

  • Beaucoup de petites opérations.
  • INSERT, UPDATE, DELETE, SELECT simples.
  • Temps réel ou quasi temps réel.
  • Optimisé pour l’exécution correcte d’actions métier.
  • Exemple : enregistrer une commande.

OLAP — analytique

  • Lecture de volumes plus importants.
  • Agrégations, comparaisons, historique, segmentation.
  • Optimisé pour la compréhension et le pilotage.
  • Exemple : CA mensuel par canal et région.
Airflow sert souvent à alimenter l’OLAP à partir de sources OLTP ou externes, sans perturber les systèmes opérationnels.
Data Warehouse

Définition opérationnelle

Définition

Un Data Warehouse est un entrepôt de données conçu pour le décisionnel. Il centralise, nettoie, historise et structure les données pour les rendre exploitables par SQL, KPI, dashboards et rapports.

Caractéristiques

  • orienté analyse ;
  • intégré depuis plusieurs sources ;
  • historisé ;
  • nettoyé et structuré ;
  • optimisé pour la lecture ;
  • gouverné par des règles partagées.
Le Data Warehouse est une logique d’organisation avant d’être un outil. PostgreSQL, BigQuery, Snowflake ou Redshift peuvent porter cette logique à des échelles différentes.
Erreur classique

Un Data Warehouse n’est pas une base de production copiée

Copie naïve

  • Tables proches des applications sources.
  • Noms de colonnes techniques ou incohérents.
  • Pas d’historique fiable.
  • Doublons ou valeurs manquantes non gérées.
  • Logique métier recalculée dans chaque dashboard.

Vraie logique DW

  • Tables pensées pour la lecture analytique.
  • Historisation et traçabilité.
  • Définitions partagées des KPI.
  • Couches intermédiaires contrôlées.
  • Calculs centralisés et testables.
Copier une source dans PostgreSQL ne suffit pas à créer un Data Warehouse.
Data Mart

Couche métier ciblée

Définition

Un Data Mart est un sous-ensemble orienté domaine, métier ou cas d’usage : ventes, finance, conformité, marketing, qualité catalogue.

Pourquoi c’est utile

  • Réduire la complexité exposée au métier.
  • Accélérer les requêtes fréquentes.
  • Stabiliser les KPI d’un domaine.
  • Donner une interface claire aux outils BI ou automatisations.
Dans un cours Airflow, une table gold ou analytics peut être vue comme un mini Data Mart alimenté par DAG.
Faits / Dimensions

Modélisation multidimensionnelle simplifiée

Table de faits

  • Contient ce qu’on mesure.
  • Exemples : quantité, montant, nombre de réservations, score, statut calculé.
  • Porte souvent les clés vers les dimensions.
  • Peut être volumineuse et historisée.

Dimensions

  • Donnent les axes d’analyse.
  • Exemples : date, produit, client, magasin, canal, catégorie.
  • Permettent de filtrer, regrouper et comparer.
  • Doivent être lisibles par les utilisateurs métier.
fait_ventes(date_id, produit_id, magasin_id, canal_id, quantite, montant) dim_date(date_id, jour, mois, trimestre, annee) dim_produit(produit_id, nom, categorie, marque)

Marketplace e-commerce : sources et usages

Sources

Un bon pipeline part de questions métier concrètes, pas d’une envie de brancher des outils.
KPI

Construire un indicateur utile

Un bon KPI doit être

  • lié à un objectif métier ;
  • mesurable sans ambiguïté ;
  • suivi dans le temps ;
  • compréhensible par le métier ;
  • actionnable si le seuil se dégrade.

Exemples qualité catalogue

  • taux de produits avec image ;
  • taux de fiches complètes ;
  • nombre de produits sans marque ;
  • score moyen de complétude ;
  • top catégories les plus dégradées ;
  • nombre d’objets sans métadonnées.
Un chiffre n’est pas automatiquement un KPI. Il devient KPI s’il aide à décider ou prioriser.

Pipeline

Définition dans une logique Data Warehouse

Pipeline de données

Chaîne d’opérations qui fait passer une donnée d’une source vers des étapes de traitement, puis vers un stockage et des usages analytiques.

Étapes typiques

  • collecte ;
  • contrôle ;
  • nettoyage ;
  • transformation ;
  • chargement ;
  • préparation à l’analyse ;
  • restitution ou action.
Dans Airflow, chaque tâche doit avoir une responsabilité claire dans cette chaîne.
ETL / ELT

Deux façons d’organiser les transformations

ApprochePrincipeAvantageLimite
ETLExtract → Transform → LoadLa cible reçoit une donnée déjà préparéePlus rigide si les besoins changent souvent
ELTExtract → Load → TransformOn conserve plus facilement le brut et on transforme dans la plateforme cibleDemande une gouvernance stricte pour éviter le désordre
Dans un cours Airflow, l’important n’est pas de défendre ETL ou ELT, mais de justifier où et pourquoi on transforme.
API

Une API n’est pas une table analytique

Ce que donne l’API

  • JSON imbriqué ;
  • champs parfois absents ;
  • pagination ;
  • formats de date variables ;
  • quotas ;
  • noms techniques ;
  • structure pensée pour le service source.

Ce que veut l’analytique

  • colonnes typées ;
  • noms métier explicites ;
  • historique ;
  • clés de jointure ;
  • indicateurs calculables ;
  • qualité mesurable ;
  • traçabilité du run.
Le travail du pipeline est de faire passer une matière première API vers une structure fiable.
JSON

Lire un schéma implicite

Identifier

  • clés stables ;
  • objets imbriqués ;
  • tableaux ;
  • types attendus ;
  • champs optionnels.

Sélectionner

  • champs métier utiles ;
  • identifiants ;
  • timestamps ;
  • liens vers fichiers ;
  • valeurs nécessaires aux KPI.

Transformer

  • flattening ;
  • conversion de type ;
  • normalisation ;
  • gestion des nulls ;
  • création de booléens.
On ne garde pas tout “au cas où” dans la table métier. On peut garder le brut ailleurs, puis structurer ce qui sert vraiment.
Data Lake

Pourquoi une base relationnelle ne suffit pas toujours

La base relationnelle gère bien

  • tables structurées ;
  • jointures ;
  • agrégations ;
  • vues SQL ;
  • requêtes analytiques classiques.

Elle gère moins naturellement

  • images ;
  • PDF ;
  • archives brutes ;
  • gros JSON variés ;
  • logs massifs ;
  • exports partenaires ;
  • objets binaires.
Une architecture moderne sépare souvent stockage brut/fichier et couche structurée analytique.
Stockage objet

S3, Azure Blob, GCS, MinIO

Principe

Le stockage objet conserve des fichiers sous forme bucket + object key. Il est très adapté aux Data Lakes car il accepte des formats variés et une organisation logique par zones.

Ce qu’on stocke

  • payload JSON brut ;
  • CSV source ;
  • images ;
  • rapports générés ;
  • exports d’anomalies ;
  • fichiers de contrôle ;
  • archives de run.
Dans le cours, MinIO peut simuler localement une logique S3 sans dépendance cloud.
Métadonnées

Un fichier sans description devient vite inutilisable

Métadonnées minimales

  • nom du fichier ;
  • bucket ;
  • object_key ;
  • source ;
  • type MIME ;
  • taille ;
  • checksum ;
  • date d’ingestion ;
  • run_id ;
  • statut de traitement.

Pourquoi c’est critique

  • retrouver un objet ;
  • relier fichier et entité métier ;
  • auditer une ingestion ;
  • rejouer un traitement ;
  • détecter les doublons ;
  • prouver ce qui a été traité.
pokemon_files(file_id, pokemon_id, bucket_name, object_key, file_type, checksum, created_at, run_id)
Lakehouse

Concilier lake et warehouse

Data Lake

  • conservation large ;
  • formats variés ;
  • brut gardé plus longtemps ;
  • souplesse pour usages futurs.

Data Warehouse

  • donnée structurée ;
  • qualité et cohérence ;
  • KPI et reporting ;
  • tables pensées pour la lecture.
Le Lakehouse cherche à garder la souplesse du Lake tout en construisant des couches fiables pour l’analyse.
Zones

Bronze, Silver, Gold : logique générale

ZoneRôleContenu typiqueAirflow doit garantir
BronzeConserver le brutJSON API, fichiers source, tables rawtraçabilité, non-perte, horodatage, run_id
SilverNettoyer et standardiserdonnées typées, dédupliquées, contrôléesqualité, schéma, règles de transformation

Les noms peuvent varier : raw/staging/curated/analytics. La logique reste la maturité progressive de la donnée.
Bronze

Zone brute : garder la preuve source

Objectif

  • conserver la donnée proche de ce qui a été reçu ;
  • permettre audit, debug et rejeu ;
  • éviter de perdre une information avant de savoir si elle sera utile ;
  • séparer collecte et interprétation.

Bonnes pratiques Airflow

  • écrire un run_id ;
  • horodater la collecte ;
  • conserver la source et les paramètres ;
  • ne pas écraser sans stratégie ;
  • journaliser volumes et erreurs ;
  • rendre le dépôt idempotent.
bronze/api_name/dt=2026-06-09/run_id=scheduled__2026-06-09T06:00:00/payload.json
Silver

Zone préparée : nettoyer sans encore faire du reporting

Transformations typiques

  • typage des colonnes ;
  • normalisation des dates ;
  • déduplication ;
  • flattening JSON ;
  • suppression ou marquage des lignes invalides ;
  • standardisation des catégories ;
  • ajout de clés techniques.

Contrôles attendus

  • schéma conforme ;
  • taux de nulls acceptable ;
  • unicité des clés ;
  • formats cohérents ;
  • volumes cohérents avec bronze ;
  • rejets tracés dans une table dédiée.
Silver n’est pas encore la couche métier finale. C’est une zone fiable de travail.
Gold

Zone métier : exposer une donnée directement exploitable

Ce qu’on y trouve

  • tables de faits ;
  • dimensions ;
  • agrégats ;
  • vues analytiques ;
  • scores métier ;
  • tables prêtes pour dashboard ou automatisation.

Exigences

  • noms compréhensibles ;
  • définitions KPI stables ;
  • historique maîtrisé ;
  • requêtes rapides ;
  • droits d’accès adaptés ;
  • pas de logique cachée dans l’outil BI.
Gold est la zone où la donnée commence vraiment à parler au métier.
Analytics

Couche analytique : pourquoi ne pas lire les tables brutes

Tables brutes

  • trop détaillées ;
  • trop proches de la source ;
  • peu lisibles ;
  • noms techniques ;
  • risque de recalculs divergents.

Vues analytiques

  • simplifient la lecture ;
  • centralisent la logique de calcul ;
  • préparent dashboards et rapports ;
  • cohérence des indicateurs ;
  • réutilisables par Metabase, n8n ou Airflow.
La qualité d’un dashboard dépend d’abord de la qualité de la couche analytique qu’il interroge.
Exemple SQL

Vue analytique qualité catalogue

CREATE OR REPLACE VIEW analytics.product_quality_metrics AS SELECT p.product_id, p.product_name, p.category, (p.image_url IS NOT NULL) AS has_image, (p.brand IS NOT NULL) AS has_brand, (f.file_id IS NOT NULL) AS has_file, ( CASE WHEN p.product_name IS NOT NULL THEN 1 ELSE 0 END + CASE WHEN p.category IS NOT NULL THEN 1 ELSE 0 END + CASE WHEN p.image_url IS NOT NULL THEN 1 ELSE 0 END + CASE WHEN p.brand IS NOT NULL THEN 1 ELSE 0 END + CASE WHEN f.file_id IS NOT NULL THEN 1 ELSE 0 END ) / 5.0 AS completeness_score FROM silver.products p LEFT JOIN silver.product_files f ON f.product_id = p.product_id;
Cette vue transforme des colonnes techniques en lecture métier : une fiche est-elle exploitable ou non ?
Qualité data

Les contrôles minimaux d’un pipeline sérieux

Complétude

  • champs obligatoires présents ;
  • taux de nulls ;
  • fichiers attendus présents.

Validité

  • types corrects ;
  • formats de dates ;
  • valeurs dans une plage attendue ;
  • référentiels connus.

Cohérence

  • clés uniques ;
  • jointures possibles ;
  • volumes attendus ;
  • pas de rupture brut → préparé.
La qualité data doit être testée dans le pipeline, pas découverte par hasard dans le dashboard.
Tables techniques

Tracer les exécutions

ingestion_runs

run_id source started_at ended_at status records_received records_inserted records_rejected error_message

Pourquoi

  • savoir ce qui a tourné ;
  • savoir combien de lignes ont été reçues ;
  • identifier les erreurs ;
  • relier données et exécution Airflow ;
  • documenter les backfills ;
  • alimenter des alertes.
Sans table de run, on a un script qui tourne. Avec table de run, on commence à avoir un pipeline observable.
Idempotence

La qualité clé pour Airflow

Définition simple

Relancer le même traitement sur le même intervalle ne doit pas créer de doublons, incohérences ou effets secondaires non maîtrisés.

Exemples de pratiques

  • clé naturelle ou clé technique stable ;
  • upsert plutôt qu’insert aveugle ;
  • partition par date ou run ;
  • suppression/rechargement contrôlé d’une partition ;
  • checksum pour fichiers ;
  • contrôle d’unicité après chargement.
Dans Airflow, le rejeu n’est pas un cas exceptionnel : c’est une fonctionnalité normale. Le pipeline doit être conçu pour ça.
Backfill

Rejouer le passé sans casser le présent

Pourquoi backfiller ?

  • corriger une transformation ;
  • charger une période manquante ;
  • changement de règle métier ;
  • migration de source ;
  • reconstruction d’un Data Mart.

Risques

  • doublons ;
  • écrasement de données valides ;
  • mélange de versions de règles ;
  • coût trop élevé ;
  • déclenchement d’alertes obsolètes.
Un bon DAG distingue la période traitée, la date d’exécution et la version de logique appliquée.
Partitions

Penser en périodes de données

Pourquoi partitionner

  • isoler les chargements par jour/mois ;
  • faciliter le rejeu ;
  • réduire les volumes relus ;
  • simplifier les contrôles ;
  • accélérer certaines requêtes.

Dans Airflow

  • utiliser data_interval_start / data_interval_end ;
  • éviter “today” non maîtrisé ;
  • passer la période aux requêtes ;
  • nommer les fichiers avec la période ;
  • contrôler les volumes par partition.
WHERE business_date >= {{ data_interval_start }} AND business_date < {{ data_interval_end }}

Airflow

Ce qu’il orchestre et ce qu’il ne doit pas cacher

Airflow orchestre

  • planification ;
  • dépendances ;
  • rejeu ;
  • monitoring des runs ;
  • retries ;
  • paramètres ;
  • connexion aux systèmes.

Airflow ne remplace pas

  • la modélisation analytique ;
  • la qualité SQL ;
  • les conventions de données ;
  • les définitions métier ;
  • la gouvernance des accès ;
  • la réflexion sur les KPI.
Un DAG propre ne compense pas une mauvaise architecture de données.
Dépendances

Ne pas tout mettre dans une seule tâche

Mauvaise approche

  • une tâche Python fait tout ;
  • debug difficile ;
  • pas de reprise intermédiaire ;
  • logs illisibles ;
  • rejeu coûteux ;
  • responsabilités mélangées.

Bonne approche

  • tâches cohérentes et séparées ;
  • contrôles entre étapes ;
  • données intermédiaires traçables ;
  • possibilité de relancer uniquement une partie ;
  • lecture claire dans l’UI.
Découper un DAG, ce n’est pas faire joli : c’est rendre le pipeline opérable.
Retries

Attention aux effets secondaires

Retry utile

  • appel API temporairement indisponible ;
  • timeout réseau ;
  • verrou transitoire ;
  • erreur technique non déterministe.

Retry dangereux si

  • insert non idempotent ;
  • fichier écrasé sans contrôle ;
  • email envoyé deux fois ;
  • alerte dupliquée ;
  • table partiellement chargée.
Avant d’activer les retries, il faut savoir ce qui se passe si la tâche est exécutée deux fois.
Secrets et config

Séparer code, paramètres et credentials

À éviter

  • URL écrite partout ;
  • ville ou catégorie hardcodée ;
  • mot de passe dans le DAG ;
  • seuil métier figé dans le code ;
  • noms de tables dupliqués.

À viser

  • Airflow Connections pour les accès ;
  • Variables ou fichiers de config pour paramètres ;
  • seuils documentés ;
  • noms centralisés ;
  • DAG réutilisable pour plusieurs périmètres.
Le code décrit la logique. La configuration adapte cette logique au contexte.
Monitoring

Observer le pipeline et pas seulement le statut vert

Statut technique

  • DAG success/failed ;
  • durée des tâches ;
  • retries ;
  • logs ;
  • SLA éventuels.

Statut data

  • nombre de lignes reçues ;
  • nombre de lignes rejetées ;
  • taux de complétude ;
  • écart vs run précédent ;
  • données manquantes ;
  • fraîcheur de la source.
Un DAG vert peut produire de mauvaises données. Le monitoring data est aussi important que le monitoring technique.
Freshness

Fraîcheur et disponibilité des données

Questions à poser

  • La source était-elle à jour ?
  • A-t-on reçu la période attendue ?
  • Le volume est-il normal ?
  • Les données sont-elles arrivées à temps pour le reporting ?

Implémentation simple

  • table de contrôle par source ;
  • max(source_updated_at) ;
  • comparaison avec data_interval ;
  • alerte si retard ;
  • KPI de fraîcheur dans analytics.
SELECT MAX(source_last_updated_at) AS last_source_update FROM silver.products;
Sécurité

Droits et exposition des données

Principe

Tout le monde n’a pas besoin de lire toutes les couches. Le métier doit souvent consommer gold/analytics, pas bronze ni technical.

Règles simples

  • limiter l’accès aux credentials ;
  • ne pas exposer le brut sensible ;
  • séparer lecture BI et écriture pipeline ;
  • tracer les exports ;
  • éviter les données personnelles inutiles dans les logs.
La gouvernance commence dès le TP : qui lit quoi, qui écrit quoi, et pourquoi ?
Visualisation

Metabase, Power BI, Superset : leur vraie place

L’outil BI lit

  • tables gold ;
  • vues analytics ;
  • agrégats ;
  • datasets préparés ;
  • requêtes SQL maîtrisées.

L’outil BI ne doit pas devenir

  • le lieu principal de nettoyage ;
  • un amas de calculs non documentés ;
  • une source de définitions KPI divergentes ;
  • un contournement de la couche analytique.
La visualisation expose une logique analytique. Elle ne doit pas remplacer la modélisation.
Automatisation

De l’indicateur à l’action

Exemple KPI

Taux de produits incomplets = 34%

incomplete_rate > 0.30

Actions possibles

  • insérer une alerte en base ;
  • envoyer un message Slack/Teams/Telegram ;
  • générer un CSV d’anomalies ;
  • déposer un rapport dans MinIO ;
  • déclencher un workflow secondaire.
Automatiser, c’est traduire une règle métier en action exécutable, visible et traçable.

Contrôles simples à demander aux étudiants

Contrôles techniques

  • le DAG écrit un run ;
  • chaque ville attendue est présente ;
  • le nombre d’observations correspond à la période ;
  • aucune tâche ne dépend d’un chemin local fragile.

Contrôles data

  • température non nulle ;
  • timestamp dans l’intervalle traité ;
  • pas de doublon ville + timestamp ;
  • valeurs météo dans des bornes plausibles ;
  • agrégat gold cohérent avec silver.
Les contrôles doivent être simples mais obligatoires : ils forcent les étudiants à penser qualité.
TP guidé

Exercice 1 — Classer les données par couche

Énoncé

  • On dispose d’un payload API brut, d’une image produit, d’une table produit nettoyée, d’un score de complétude, d’un dashboard qualité, d’un fichier CSV partenaire.
  • Classer chaque élément en bronze, silver, gold ou analytics, puis justifier.

Attendu

  • Bronze : payload, CSV brut, image.
  • Silver : table produit nettoyée, métadonnées fichier.
  • Gold : score de complétude consolidé.
  • Analytics : vue KPI ou dashboard.
Objectif : vérifier qu’ils comprennent le niveau de maturité de la donnée.
TP guidé

Exercice 2 — Dessiner un DAG data warehouse

Contraintes

  • source API ;
  • conservation du brut ;
  • transformation silver ;
  • contrôles qualité ;
  • chargement gold ;
  • alerte si KPI dégradé ;
  • table ingestion_runs obligatoire.

Critères de correction

  • découpage clair des tâches ;
  • dépendances cohérentes ;
  • rejeu possible ;
  • traçabilité ;
  • séparation source / métier / analytics ;
  • pas de tâche “do_everything”.
Cet exercice prépare directement la conception des DAGs Airflow.

Erreurs fréquentes

Ce que tu dois corriger explicitement en cours

Erreurs de conception

  • confondre raw et analytics ;
  • charger directement un JSON API dans une table métier ;
  • ne pas historiser ;
  • ne pas tracer les runs ;
  • créer des KPI dans chaque dashboard ;
  • stocker des fichiers sans métadonnées.

Erreurs Airflow

  • utiliser datetime.now() pour la période traitée ;
  • insert aveugle non idempotent ;
  • tâche unique trop grosse ;
  • credentials dans le code ;
  • retry qui duplique les effets ;
  • aucun contrôle après chargement.
Ces erreurs sont plus importantes à corriger que des détails de syntaxe Airflow.
Mini évaluation

Questions rapides de compréhension

Questions

  • Pourquoi une API ne fournit-elle pas directement une donnée analytique ?
  • Quelle différence entre bronze et silver ?
  • Pourquoi une table ingestion_runs est-elle utile ?
  • Pourquoi lire les tables brutes dans Metabase est une mauvaise idée ?
  • Que signifie idempotence dans un DAG ?

Réponses attendues

  • source ≠ modèle métier ;
  • bronze conserve, silver prépare ;
  • tracer, auditer, rejouer ;
  • calculs instables et peu lisibles ;
  • relancer sans doublonner ni corrompre.
Ces questions peuvent servir de note de compréhension de cours.

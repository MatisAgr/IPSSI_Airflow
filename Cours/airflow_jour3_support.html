Un workflow fiable ne réussit pas seulement une fois : il reste lisible, sûr et exploitable dans le temps.

Fiabilité

Un workflow robuste ne se limite pas à “ça passe une fois”

Ce qu’un workflow fiable doit supporter

  • un échec ponctuel sur une dépendance externe ;
  • une relance automatique d’une tâche ;
  • une réexécution d’un run déjà identifié ;
  • une lecture claire de ce qui a été fait ;
  • une cohérence de bout en bout entre extraction, transformation et chargement.

Ce qu’il faut éviter

  • un succès technique qui masque un résultat incomplet ;
  • des doublons créés par un rerun mal pensé ;
  • une tâche qui écrit partiellement puis laisse la cible dans un état ambigu ;
  • une relance qui reproduit la même erreur sans réflexion ;
  • un pipeline dont on ne peut pas expliquer le résultat produit.
La robustesse ne concerne pas seulement la tâche isolée. Elle concerne aussi la cohérence logique du pipeline complet.
Incidents temporaires

Retry, retry delay et timeout

Retry

Nouvelle tentative automatique après un échec.

  • incident réseau ponctuel ;
  • API momentanément indisponible ;
  • verrou temporaire ;
  • fluctuation courte d’un système externe.

Retry delay

Délai entre deux tentatives successives.

  • évite de répéter immédiatement le même échec ;
  • laisse du temps à la ressource externe pour revenir ;
  • stabilise les comportements sur incident passager.

Timeout

Limite la durée d’une tâche pour éviter l’attente infinie.

  • appel API bloqué ;
  • requête SQL interminable ;
  • traitement figé ;
  • dépendance externe lente ou cassée.
Le retry aide à gérer l’instable. Le timeout évite qu’une tâche attende sans fin. Aucun des deux ne corrige une mauvaise conception.
Diagnostic

Échec temporaire, échec structurel, retry et rerun

Échec temporaire

  • API momentanément indisponible ;
  • latence réseau ;
  • ressource provisoirement saturée ;
  • erreur transitoire d’infrastructure ;
  • verrou temporaire.

Dans ces cas, retry et retry delay peuvent suffire.

Échec structurel

  • logique métier erronée ;
  • schéma inattendu ;
  • colonne absente ;
  • credentials incorrects ;
  • bug de code ou table cible mal définie.

Ici, relancer plusieurs fois ne résout rien : il faut corriger la cause.

Retry

Nouvelle tentative automatique après échec.

Rerun

Réexécution volontaire ou technique d’une tâche ou d’un run déjà exécuté.

  • risque de doublons ;
  • second chargement des mêmes lignes ;
  • écrasement incohérent ;
  • traçabilité confuse si la stratégie n’est pas claire.
Idempotence

Exécuter plusieurs fois sans dégrader le résultat

Définition

Une tâche idempotente produit le même résultat logique si elle est exécutée plusieurs fois dans les mêmes conditions.

Cette propriété devient critique en cas de retry ou de rerun.

Pourquoi c’est central

  • sécuriser les relances ;
  • éviter les doublons ;
  • préserver la cohérence métier ;
  • garder une cible compréhensible ;
  • faciliter l’exploitation dans la durée.

Exemples idempotents

  • réécrire proprement une partition donnée ;
  • faire un upsert cohérent ;
  • charger avec une clé unique empêchant le doublon ;
  • recalculer un indicateur à partir de la source.

Exemples non idempotents

  • insérer la même donnée à chaque relance ;
  • accumuler des résultats sans logique de déduplication ;
  • écrire un résultat partiel puis le compléter ;
  • append systématique sans contrôle.
Un pipeline mature doit rester sûr quand on le retente et quand on le rejoue.
Intégrité

Une tâche doit se comporter comme une transaction

Comportement attendu

  • réussir proprement ou échouer proprement ;
  • ne pas laisser de résultat semi-cassé ;
  • ne pas rendre exploitable un chargement incomplet ;
  • protéger l’intégrité du pipeline global.

Pourquoi “success” ne suffit pas

  • le résultat peut être incomplet ;
  • la source traitée peut être mauvaise ;
  • le mauvais intervalle temporel peut avoir été utilisé ;
  • des doublons peuvent avoir été créés ;
  • un traitement techniquement réussi peut être faux fonctionnellement.
Un run réussi techniquement n’est pas automatiquement un run correct du point de vue métier.
Contrôle qualité

Avant d’accepter un run, il faut décider si la donnée est digne de confiance

Pourquoi contrôler la donnée

  • une API peut répondre, mais renvoyer une donnée incomplète ;
  • le code peut tourner, mais produire une information incohérente ;
  • la base peut recevoir des lignes, mais des lignes inutilisables ;
  • une réussite technique ne garantit pas une réussite fonctionnelle.

Question à poser à chaque run

La donnée produite est-elle acceptable, fiable et exploitable ?

Le contrôle qualité ne mesure pas seulement si le pipeline fonctionne. Il mesure si le résultat mérite d’être accepté.

Complétude

Les champs attendus sont-ils présents ?

Unicité

Y a-t-il des doublons interdits ?

Cohérence

Les valeurs ont-elles du sens ensemble ?

Fraîcheur

La donnée est-elle assez récente ?

Structure

Les champs attendus existent-ils encore ?

Règles de validation

Complétude, unicité, cohérence, fraîcheur et présence des champs

Complétude

  • la ville est renseignée ;
  • la date de mesure existe ;
  • la température n’est pas vide ;
  • les champs indispensables au métier sont présents.

Unicité

  • une ligne par ville et date ;
  • pas deux mesures identiques pour le même intervalle ;
  • pas de doublon sur un identifiant de référence.

Cohérence

  • température dans une plage raisonnable ;
  • précipitations non négatives ;
  • date compatible avec le run traité ;
  • valeurs compatibles entre elles.

Fraîcheur

  • la donnée n’est pas trop ancienne ;
  • la dernière mesure correspond à la fréquence d’ingestion attendue.

Présence des champs attendus

  • une clé JSON attendue n’a pas disparu ;
  • un bloc de réponse est toujours fourni ;
  • le schéma source n’a pas dérivé sans être pris en compte.
Erreurs

Erreur technique et erreur de qualité ne désignent pas le même problème

Erreur technique

  • timeout ;
  • problème réseau ;
  • table inaccessible ;
  • erreur Python ;
  • SQL invalide.

Le pipeline n’arrive pas à exécuter une opération.

Erreur de qualité

  • doublons ;
  • valeurs nulles interdites ;
  • date trop ancienne ;
  • format inattendu ;
  • nombre de lignes anormal.

Le pipeline tourne, mais la donnée obtenue n’est pas acceptable.

Que faire si la qualité échoue

  • bloquer le chargement final ;
  • déclencher une branche d’alerte ;
  • écrire dans une table d’anomalies ;
  • journaliser clairement l’échec ;
  • marquer le run comme non exploitable.

Place du contrôle qualité

  • après extraction ;
  • après transformation ;
  • avant chargement final ;
  • ou après chargement pour une vérification complémentaire.

La qualité n’est pas un bonus décoratif placé à la fin : elle fait partie du workflow.

Supervision

L’UI, les logs et l’historique des runs font partie de la valeur d’Airflow

UI Airflow

  • voir les DAGs ;
  • voir les runs ;
  • voir les tâches et leurs états ;
  • lire les logs ;
  • diagnostiquer plus vite.

Logs

  • ce qu’une tâche a fait ;
  • avec quels paramètres ;
  • à quel moment ;
  • où elle a échoué ;
  • quel message d’erreur a été produit.

États et historique

  • tâche en attente, exécutée, ignorée ou échouée ;
  • lecture des dépendances amont ;
  • stabilité ou récurrence des incidents ;
  • impact d’une correction dans le temps.
Un pipeline devient réellement exploitable lorsqu’il devient observable.
TP 3A — Fin de matinée

Ajouter robustesse et contrôle qualité

Objectif

Faire évoluer le pipeline Open-Meteo réalisé la veille pour lui ajouter des mécanismes de fiabilité et de contrôle.

Travail demandé

  • 1Ajouter des retries au DAG ou aux tâches pertinentes.
  • 2Ajouter un timeout sur au moins une tâche.
  • 3Créer une tâche de contrôle qualité.
  • 4Empêcher le chargement final si les données sont invalides.
  • 5Prévoir une sortie d’erreur lisible.

Consignes

  • ne pas ajouter des retries partout sans justification ;
  • distinguer les échecs qui méritent un retry de ceux qui relèvent d’un problème de logique ;
  • choisir des contrôles qualité simples mais justifiés ;
  • garder un DAG lisible ;
  • montrer clairement le chemin nominal et le chemin d’échec.

Livrables attendus

  • DAG enrichi ;
  • description des contrôles qualité retenus ;
  • preuve de l’exécution ;
  • courte explication des choix de robustesse.
Workflow conditionnel

Un DAG mature ne suit pas toujours une ligne unique

Branchement conditionnel

Le workflow suit un chemin différent selon une condition observée pendant l’exécution.

  • si la qualité est correcte, on charge ;
  • sinon, on alerte ;
  • si aucune donnée n’est reçue, on stoppe ;
  • si un seuil d’erreur est dépassé, on rejette.

Logique métier

  • le branching traduit une règle métier ou opérationnelle ;
  • il relie détection d’un état et comportement du workflow ;
  • il rend explicite la réaction du pipeline à une anomalie.

Le mécanisme technique n’a de sens que s’il porte une règle compréhensible.

Un bon DAG ne décrit pas seulement la réussite. Il décrit aussi le comportement en cas de problème.
Chemins

Chemin nominal, chemin d’alerte et règles de déclenchement

Chemin nominal

  • extraction ;
  • transformation ;
  • validation ;
  • chargement ;
  • fin normale.

Chemin d’alerte

  • arrêt du chargement ;
  • écriture d’une anomalie ;
  • notification ;
  • orientation vers une sortie dédiée.

Règles possibles

  • nombre de lignes reçues = 0 ;
  • champ obligatoire absent ;
  • doublons détectés ;
  • fraîcheur non respectée ;
  • taux de données manquantes au-dessus d’un seuil.
Structuration

Un projet Airflow maintenable ne se résume pas à un fichier DAG qui fonctionne

Un DAG lisible

  • se comprend rapidement ;
  • montre des étapes nommées clairement ;
  • expose une logique visible ;
  • évite les croisements inutiles ;
  • sépare les responsabilités.

Des tâches compréhensibles

  • une responsabilité claire ;
  • une sortie identifiable ;
  • une relance propre ;
  • un rôle lisible dans le workflow.

Le but n’est pas de créer des micro-tâches absurdes, mais des tâches nettes et maintenables.

Séparation Orchestration / logique métier

Le DAG orchestre. La logique métier détaillée vit idéalement dans des fonctions, modules ou scripts dédiés.

Séparation Orchestration / SQL

Les requêtes peuvent être isolées, nommées et organisées pour éviter un DAG illisible rempli de SQL embarqué.

Réutilisabilité

Une structure propre permet d’adapter une source, une configuration ou un workflow sans tout réécrire.

Projet maintenable

Conventions, modularité et ouverture vers la production

Conventions minimales

  • noms de tâches explicites ;
  • conventions de dossiers ;
  • conventions sur les fonctions ;
  • conventions sur les paramètres ;
  • conventions de logs et de traçabilité.

Ce qu’un projet d’équipe ajouterait

  • plus de modularité ;
  • plus de tests ;
  • séparation plus stricte des environnements ;
  • monitoring plus riche ;
  • gestion plus propre des secrets ;
  • industrialisation du déploiement.

Scheduler

Surveille les DAGs et décide quand les tâches peuvent partir.

Executor

Correspond à la manière dont les tâches sont effectivement exécutées.

Production

On ne raisonne plus comme sur une démo locale : concurrence, stabilité, logs, monitoring et déploiement deviennent centraux.

Un bon projet Airflow se juge à sa lisibilité, à son découpage et à sa capacité à grandir proprement.

Synthèse

Ce qu’il faut retenir avant le projet intégrateur

Fiabilité et qualité

  • retry, retry delay et timeout gèrent surtout les incidents temporaires ;
  • un rerun impose de penser l’idempotence ;
  • un pipeline peut réussir techniquement et échouer fonctionnellement ;
  • complétude, unicité, cohérence, fraîcheur et présence des champs sont des contrôles classiques ;
  • un run ne doit pas être accepté sans validation explicite de la donnée.

Exploitation et structuration

  • l’UI Airflow, les logs et l’historique des runs font partie du monitoring ;
  • un workflow peut suivre plusieurs chemins selon une règle ;
  • il faut distinguer chemin nominal, chemin d’alerte et rejet ;
  • un projet maintenable repose sur un DAG lisible et des responsabilités séparées ;
  • un pipeline exploitable est un pipeline observable, réutilisable et cohérent dans le temps.
Le mercredi, on passe d’un pipeline qui fonctionne à un pipeline qu’on peut exploiter avec confiance.

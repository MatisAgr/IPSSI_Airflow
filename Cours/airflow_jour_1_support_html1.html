Avant l’outil

Pourquoi commencer par le problème avant de parler d’Airflow ?

Point de départ

Une équipe doit exécuter régulièrement plusieurs traitements : récupérer des données, les transformer, les charger, vérifier le résultat et alerter en cas d’échec.

  • Le besoin réel n’est pas “faire tourner un script”.
  • Le besoin réel est de fiabiliser une chaîne de traitements.
  • La question centrale devient : quoi lancer, quand, dans quel ordre, avec quelle visibilité ?

Pourquoi c’est décisif

  • Sans problème clairement identifié, Airflow ressemble à une usine à gaz.
  • Avec le bon problème, Airflow devient un outil de structure et de pilotage.
  • On ne cherche pas seulement une exécution automatique, mais une exécution compréhensible, supervisable et rejouable.

Du plus simple au plus robuste

Script manuel, cron, orchestrateur : trois niveaux très différents

Script manuel

Ce que l’on gagne

  • Simple à démarrer
  • Rapide pour un besoin ponctuel
  • Peu de mise en place

Ce que l’on perd

  • Dépendance à une personne
  • Oublis et horaires variables
  • Faible historique et faible traçabilité
Cron

Ce que cron fait bien

  • Lancer une commande à heure fixe
  • Rester léger et simple
  • Traiter des cas isolés et basiques

Ce que cron gère mal

  • Dépendances complexes
  • Supervision globale
  • Reprise et lisibilité des échecs
Orchestrateur

Ce qu’il apporte

  • Ordre d’exécution maîtrisé
  • Gestion des dépendances
  • Historique, logs, supervision
  • Possibilité de rejouer
  • Lecture centralisée des workflows
Cron répond surtout à la question : quand lancer une commande ? Un orchestrateur répond aussi à : quoi lancer, dans quel ordre, avec quel état, et que faire si cela échoue ?
Le vrai problème

Ce que ni le manuel ni cron ne modélisent proprement : l’enchaînement du workflow

Dès que le pipeline devient sérieux, il faut gérer :

  • des dépendances entre étapes ;
  • des branches de traitement ;
  • des cas d’échec ;
  • des logs exploitables ;
  • des exécutions régulières ;
  • plusieurs workflows et parfois plusieurs équipes.
Plus il y a de dépendances, de reprises, de supervision et de fréquence, plus on a besoin d’un orchestrateur.
Définition

Qu’est-ce que l’orchestration ? Qu’est-ce qu’Airflow ?

Orchestration

L’orchestration consiste à piloter des traitements dans le temps et dans un ordre maîtrisé.

Définir
Planifier
Exécuter
Surveiller
Rejouer

Airflow

Airflow est une plateforme d’orchestration de workflows. Un workflow y est représenté sous la forme d’un DAG, c’est-à-dire un graphe orienté sans cycle.

  • graphe : des étapes reliées ;
  • orienté : il y a un sens d’exécution ;
  • sans cycle : on évite la boucle infinie dans le workflow.
Airflow ne décrit pas seulement un traitement. Il décrit aussi sa logique d’exécution.
Workflow as code

Pourquoi cette logique est importante en data engineering

Versionnable

Le workflow vit dans du code, donc dans Git, avec historique et comparaison.

Relisible

Le pipeline n’est pas caché dans un flux opaque ; il peut être relu et compris.

Testable

On peut isoler et contrôler certaines briques du workflow.

Maintenable

Les évolutions deviennent plus propres qu’avec une accumulation de scripts dispersés.

Ce qu’Airflow est

  • un orchestrateur ;
  • un planificateur ;
  • un superviseur ;
  • un outil de visualisation des workflows ;
  • un point central de lecture des runs et des états.

Ce qu’Airflow n’est pas

  • une base de données ;
  • un moteur BI ;
  • un data warehouse ;
  • un outil magique de qualité des données ;
  • un remplaçant de la logique métier.
Airflow n’améliore pas une mauvaise logique métier. Il rend visible et pilotable la logique qu’on lui donne.
Première modélisation

Le DAG est le cœur du raisonnement Airflow

Définition utile

Un DAG représente un workflow : ses tâches, leur ordre, leur planification et certaines règles d’exécution.

  • Il ne représente pas seulement un script.
  • Il représente une organisation du traitement.
  • Il donne une lecture claire de l’enchaînement et des dépendances.

Exemples de tâches dans un DAG data

  • appeler une API ;
  • parser un JSON ;
  • charger une table SQL ;
  • calculer une synthèse ;
  • envoyer une alerte.
Concepts fondamentaux

DAG, task, task instance, DAG run, scheduler, UI

DAG

Le modèle global du workflow : tâches, ordre, dépendances, planning.

Task

Une unité de travail identifiable dans le workflow.

Task instance

L’exécution concrète d’une tâche dans un run donné.

DAG run

L’occurrence concrète du workflow à un instant logique donné.

Scheduler

Le composant qui détermine quels runs et quelles tâches peuvent partir.

UI

L’interface qui expose les DAGs, runs, états, tâches et logs.

Si ces notions sont confondues, l’interface Airflow devient difficile à lire et le comportement du workflow devient difficile à comprendre.
Comprendre l’exécution

Task ≠ task instance, DAG ≠ DAG run

Définition vs exécution

  • fetch_weather = la définition de la tâche ;
  • fetch_weather exécutée pour le run du jour = la task instance ;
  • le DAG = le modèle général ;
  • le DAG run = l’occurrence concrète du workflow dans le temps.

États utiles à lire dans Airflow

success failed running queued skipped upstream_failed

Une tâche n’est pas seulement “lancée” ou “non lancée”. Son état participe à la lecture globale du workflow et au diagnostic d’incident.

Pilotage temporel

Le scheduler et l’UI font partie de la valeur d’Airflow

Rôle du scheduler

  • surveiller quels DAGs existent ;
  • déterminer quels runs doivent être créés ;
  • vérifier quelles tâches peuvent partir ;
  • respecter planning et dépendances.
Le scheduler ne fait pas qu’attendre l’heure. Il pilote la possibilité d’exécution.

Rôle de l’interface

  • voir les DAGs ;
  • voir les runs ;
  • voir les tâches ;
  • voir les états ;
  • voir les logs ;
  • piloter certains comportements.

L’interface n’est pas un gadget. Elle transforme le workflow en objet visible et exploitable.

Avant d’écrire un DAG, il faut apprendre à penser un DAG. Les erreurs les plus courantes viennent d’un mauvais découpage du workflow : tâche fourre-tout, dépendances artificielles, oubli du chemin d’échec.

Reprise de l’après-midi

Les concepts fondamentaux d’Airflow

Objectif

Donner des définitions propres et stables pour lire correctement un workflow Airflow, comprendre son exécution et interpréter ce que montre l’interface.

Pourquoi c’est indispensable

  • comprendre la différence entre le modèle et l’exécution ;
  • lire les états et les logs sans confusion ;
  • interpréter correctement le comportement d’un DAG ;
  • préparer l’écriture d’un premier workflow propre.
Si vous ne distinguez pas DAG, task, task instance et Dag Run, vous ne comprendrez jamais vraiment ce que vous voyez dans l’interface, ni pourquoi un workflow se comporte comme il se comporte.
Définitions de base

DAG, task, task instance, Dag Run

DAG

  • modèle du workflow ;
  • organise tâches, dépendances, planning et règles associées.

Task

  • unité de travail dans le workflow.

Task Instance

Une tâche, c’est la définition. Une task instance, c’est l’exécution concrète de cette tâche dans un run donné.

Exemple :

  • tâche = fetch_weather
  • task instance = fetch_weather exécutée pour un run précis

Dag Run

Le DAG est le modèle général. Le Dag Run est l’occurrence concrète de ce workflow à un instant logique donné.

États et pilotage

États d’une tâche, états d’un Dag Run, rôle du scheduler et de l’UI

États d’une tâche

success failed running queued skipped upstream_failed

Une tâche n’est pas juste exécutée ou non. Elle a un état, et cet état compte dans le pilotage du workflow.

États d’un Dag Run

  • si les feuilles terminales aboutissent selon la logique prévue, le run peut être considéré comme réussi ;
  • sinon, il échoue.

Point subtil : la structure du DAG influence la lecture du succès global.

Scheduler

  • il regarde quels DAGs existent ;
  • il regarde quels runs doivent être créés ;
  • il vérifie quelles tâches peuvent partir ;
  • il respecte dépendances et planning.
Le scheduler ne fait pas qu’attendre l’heure. Il pilote la possibilité d’exécution.

UI Airflow

  • voir les DAGs, les runs, les tâches et les états ;
  • voir les logs ;
  • piloter certains comportements.
L’interface n’est pas un gadget. Elle fait partie de la valeur d’Airflow, parce qu’elle apporte visibilité et supervision.
Conception de workflow

Penser un DAG correctement

Un DAG n’est pas un organigramme métier brut

Un processus métier et un DAG ne sont pas exactement la même chose.

  • Le métier dit : récupérer les données, les traiter, produire un résultat.
  • Le DAG doit transformer cela en unités de travail, dépendances, règles d’échec et logique de reprise.

Bonne granularité d’une tâche

  • une responsabilité identifiable ;
  • une sortie compréhensible ;
  • une valeur logique dans le workflow.

Trop gros

  • appel API ;
  • parsing ;
  • nettoyage ;
  • chargement SQL ;
  • notification.

Conséquences : logs peu utiles, reprise difficile, faible lisibilité, responsabilités mélangées.

Trop fin

Découper en 25 micro-tâches pour des opérations triviales.

Conséquences : DAG illisible, maintenance lourde, bruit inutile, complexité artificielle.

Dépendances

Séquentiel, parallèle, dépendance utile, dépendance artificielle

Séquentiel vs parallèle

  • récupérer météo Paris ;
  • récupérer météo Berlin ;
  • récupérer météo Madrid.

Ces appels peuvent être parallèles. En revanche, le chargement d’une table de synthèse dépend de la récupération précédente.

Un bon DAG reflète les dépendances réelles, pas les habitudes de code du développeur.

Dépendance utile vs dépendance artificielle

  • transform_json dépend de fetch_api : logique.
  • send_alert dépend de load_db si alerte de succès : logique.
  • cleanup_tmp_files qui dépend de tout sans nécessité claire : peut-être artificiel.
Chaque flèche du DAG doit se justifier.
Erreurs classiques

Ce qui rend un DAG fragile, illisible ou difficile à maintenir

Erreurs de conception

  • La tâche fourre-tout : tout mettre dans une seule tâche Python.
  • Le DAG spaghetti : trop de branches, trop de croisements, aucune lisibilité.
  • La dépendance pour rien : forcer un ordre parce que cela rassure, pas parce que c’est nécessaire.
  • Mélanger orchestration et logique métier : le DAG pilote le workflow, il ne contient pas toute la logique profonde au mauvais endroit.
  • Ignorer l’échec : penser seulement au chemin nominal.

Séparer les rôles

  • Airflow orchestre ;
  • Python ou SQL exécute une logique ;
  • PostgreSQL stocke ;
  • l’API fournit une donnée source.
Le DAG doit être le chef d’orchestre, pas l’orchestre complet.

Lisibilité et maintenabilité

  • autre développeur ;
  • futur toi ;
  • exploitation ;
  • correction d’incident ;
  • évolutions.
Le vrai test d’un DAG n’est pas seulement : est-ce qu’il tourne ? Le vrai test, c’est : est-ce qu’un autre humain peut comprendre rapidement ce qu’il fait ?

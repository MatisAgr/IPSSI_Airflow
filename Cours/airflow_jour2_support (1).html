Scheduling

Exécution manuelle, exécution planifiée, schedule, cron et presets

Exécution manuelle

  • tester ;
  • rejouer ;
  • vérifier un comportement ;
  • lancer un flux à la demande.

Exécution planifiée

  • tous les jours ;
  • toutes les heures ;
  • tous les lundis ;
  • tous les premiers jours du mois.

Le schedule

Définit la logique de planification du DAG : presets, cron, durées, timetables.

Cron

0 6 * * *
0 * * * *
0 0 * * 1

Cron décrit quand on déclenche, pas forcément la période de données traitée.

Presets utiles

@hourly
@daily
@weekly
@monthly

Ce n’est pas parce qu’un DAG est quotidien que sa logique temporelle est triviale.
Temps logique

Data interval, date logique, start_date et catchup

Data interval

  • un DAG run représente une période de données ;
  • il ne représente pas seulement un instant d’exécution ;
  • un run quotidien peut couvrir une fenêtre complète de données.

Date logique

La date affichée ne doit pas être lue comme “le moment où ça tourne”, mais comme un repère logique lié à la période de données.

Pourquoi la confusion est fréquente

  • confusion entre exécution réelle et date logique ;
  • lecture du calendrier comme un simple agenda ;
  • oubli qu’un run quotidien démarre souvent après la fin de la période qu’il couvre.

Start date et catchup

  • start_date ancre le premier intervalle de données ;
  • catchup=True rattrape les intervalles passés ;
  • catchup=False se concentre généralement sur le plus récent.
Airflow raisonne en logique de période traitée, pas seulement en logique “je lance quelque chose maintenant”.
Point difficile

Exécuter tous les jours n’est pas la même chose que traiter la période du jour

Logique calendrier

  • quand le DAG est déclenché ;
  • à quelle fréquence il part ;
  • quelle expression cron ou quel preset est utilisé.

Logique data

  • quelle période ce run représente ;
  • quelles données doivent être disponibles ;
  • que faire des périodes passées ;
  • comment rejouer proprement.
Cron répond à “quand lancer ?”. Airflow doit aussi permettre de répondre à “quelle période de données ce run représente-t-il ?”.
API et données externes

Une API n’est pas une table analytique : c’est une source à interpréter

Ce qu’est une API dans un pipeline

Une API est une interface par laquelle un système expose des données ou des services à un autre système. Dans un pipeline data, elle joue souvent le rôle de source externe.

  • météo ;
  • taux de change ;
  • catalogue produit ;
  • données logistiques ;
  • données publiques.

Donnée source vs donnée exploitable

  • la donnée source est fournie selon la logique du système externe ;
  • la donnée exploitable est sélectionnée, nettoyée, convertie et structurée pour un usage métier.
Une API ne fournit pas directement une table analytique. Elle fournit une matière première.
Lire un JSON

Schéma implicite, champs utiles et difficultés fréquentes

Ce que l’on trouve dans un JSON

  • clés simples ;
  • objets imbriqués ;
  • tableaux ;
  • champs présents ou absents selon les cas.

Questions à se poser

  • quelles informations sont réellement utiles ?
  • quelles clés sont stables ?
  • quelles données sont techniques ?
  • quelles données sont métier ?

Schéma implicite

  • noms de champs ;
  • types attendus ;
  • cardinalités ;
  • présence ou absence de certaines propriétés.

Problèmes fréquents

  • valeurs manquantes ;
  • structures imbriquées ;
  • horodatages et timezones ;
  • champs inutiles ;
  • évolution de la réponse API.

Sélection des champs

Dans un pipeline, on ne sélectionne pas tout par réflexe. On sélectionne ce qui a une utilité pour le besoin métier, la future table cible ou la traçabilité.

La lecture d’une API n’est pas un acte purement technique. C’est un travail d’interprétation de la donnée source.

Du brut vers le structuré

Extraction, transformation, chargement

Extraction

  • API ;
  • fichier ;
  • base source ;
  • service externe.

Transformation

  • sélection ;
  • conversion ;
  • nettoyage ;
  • restructuration ;
  • harmonisation.

Chargement

  • table de staging ;
  • table métier ;
  • table analytique ;
  • table de suivi.
Charger ne veut pas dire “copier-coller la source dans la base”.
Couches de données

Brut, intermédiaire, table cible, table d’ingestion et traçabilité

Brut

  • proche de la source ;
  • parfois désordonné ;
  • utile pour audit, debug et reprise.

Intermédiaire

  • version de travail ;
  • partiellement nettoyée ;
  • encore technique dans certains cas ;
  • utile pour séparer les étapes.

Table cible

  • stable ;
  • lisible ;
  • cohérente avec le besoin ;
  • interrogeable proprement.

Table d’ingestion

  • source ;
  • date de début et date de fin ;
  • statut ;
  • nombre de lignes reçues ;
  • nombre de lignes insérées ;
  • erreurs éventuelles ;
  • run concerné.

Table métier et traçabilité

  • donnée utile pour l’usage final ;
  • structure lisible et réutilisable ;
  • capacité à répondre à : quand, avec quelle source, combien, où cela a bloqué, quel run a produit la donnée.
Gouvernance minimale

Pourquoi éviter le hardcode et séparer code, configuration et paramètres

Le hardcode pose problème

  • ville écrite en dur ;
  • URL écrite en dur ;
  • seuil écrit en dur ;
  • credentials dans le code ;
  • nom de table figé partout.

Conséquences : faible réutilisabilité, maintenance pénible, duplication, erreurs lors des changements, sécurité médiocre.

Deux familles de paramètres

  • paramètres métier : villes, fréquence métier, seuil d’alerte, mesures à récupérer ;
  • paramètres techniques : connexion base, endpoint API, timeout, retries, schéma, bucket.
Le code décrit la logique. La configuration adapte cette logique à un contexte.
Réutilisabilité

Variables, connexions, documentation minimale et nommage propre

Variables

Pratiques pour stocker certaines valeurs de configuration, mais à utiliser avec discernement.

Connexions

Centralisent les accès à PostgreSQL, API, S3 et autres systèmes externes.

Documentation minimale

Rôle du DAG, source, cible, fréquence, logique de traitement, paramètres importants.

Nommage

Doit permettre de comprendre le DAG, lire l’UI et diagnostiquer plus vite.

Ce qu’un DAG propre doit rester

  • lisible ;
  • propre ;
  • configurable ;
  • réutilisable ;
  • testable dans une certaine mesure.

Conclusion de la séquence

Un pipeline utile n’est pas seulement un pipeline qui marche. C’est un pipeline qu’on peut relire, adapter, relancer, maintenir et faire évoluer sans repartir de zéro.

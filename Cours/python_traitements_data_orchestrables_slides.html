I.a — Rôle de Python

Python fait le travail ; l’orchestrateur organise le travail

Python répond à

  • comment récupérer une donnée ;
  • comment la transformer ;
  • comment calculer un indicateur ;
  • comment écrire le résultat ;
  • comment signaler un problème métier.

L’orchestrateur répond à

  • quand lancer le traitement ;
  • dans quel ordre ;
  • que faire si une étape échoue ;
  • comment voir les logs ;
  • comment relancer proprement.
Exemple : Python nettoie le CSV de ventes et charge les KPI ; l’orchestrateur déclenche le traitement chaque matin, vérifie l’ordre et conserve l’historique.
I.b — Séparation des responsabilités

Un bon traitement Python doit pouvoir vivre hors de l’outil qui l’exécute

Erreur classique

  • mettre toute la logique dans le DAG ;
  • écrire des fonctions longues directement dans l’orchestrateur ;
  • embarquer SQL, chemins, paramètres et règles métier partout ;
  • rendre le traitement difficile à tester hors outil.

Bon réflexe

  • écrire un code Python autonome ;
  • le rendre lançable indépendamment ;
  • l’appeler ensuite depuis Airflow, cron, n8n, un job cloud ou une CLI ;
  • protéger le projet si l’orchestrateur change.
Orchestrateur = déclencher, ordonner, superviser
Python = traiter, transformer, contrôler, charger
I.c — Script data fragile

Un script qui marche une fois n’est pas un composant professionnel

import pandas as pd
import requests

url = "https://api.exemple.com/sales"
r = requests.get(url)
data = r.json()

df = pd.DataFrame(data["results"])
df["date"] = pd.to_datetime(df["date"])
df["amount"] = df["amount"].astype(float)
df = df[df["amount"] > 0]

df.to_csv("C:/Users/mathieu/Desktop/result.csv", index=False)
print("done")
I.d — Job data autonome

Un job professionnel a un contrat d’exécution clair

Il doit avoir

  • une entrée claire ;
  • une sortie claire ;
  • une configuration explicite ;
  • une gestion des erreurs ;
  • des logs exploitables ;
  • un comportement reproductible ;
  • une documentation minimale ;
  • des tests sur les parties critiques.

Questions à poser

  • qu’est-ce qu’il lit ?
  • qu’est-ce qu’il produit ?
  • quels paramètres utilise-t-il ?
  • comment sait-on qu’il a réussi ?
  • comment sait-on pourquoi il a échoué ?
Mini-exercice oral : météo, fichier clients, KPI de ventes — identifier entrée, sortie, paramètres, erreur bloquante et logs attendus.
II.a — Du script au projet

Un projet organisé répond à “comment ce traitement peut-il vivre dans le temps ?”

Script unique

  • acceptable pour une démonstration ;
  • utile en exploration ;
  • rapide pour un prototype ;
  • devient vite fragile en maintenance.

Projet organisé

  • sépare les responsabilités ;
  • rend le code lisible ;
  • permet de tester sans tout exécuter ;
  • facilite le remplacement d’une source ou cible ;
  • rend l’exécution automatisable.
Un script répond : “comment je fais marcher ça maintenant ?” Un projet répond : “comment ce traitement peut-il vivre dans le temps ?”
II.b — Structure minimale

Organisation recommandée pour un projet Python data

data_job/
  README.md
  pyproject.toml
  .env.example
  src/
    data_job/
      __init__.py
      config.py
      extract.py
      transform.py
      validate.py
      load.py
      main.py
  tests/
    test_transform.py
    test_validate.py

Message important

La structure n’est pas là pour faire joli : elle force à clarifier les responsabilités.

  • README.md explique lancement et paramètres ;
  • .env.example documente la configuration ;
  • src/ contient le code applicatif ;
  • tests/ contient les tests unitaires.
II.c — Découpage par responsabilité

Chaque module doit avoir un rôle clair

Extraction

extract.py récupère la matière première. Il ne nettoie pas profondément les données.

Transformation

transform.py prend une donnée récupérée et la transforme. Il n’appelle pas l’API.

Validation

validate.py vérifie que la donnée est acceptable. Il ne charge pas en base.

def run():
    config = load_config()
    raw_data = extract_sales(config)
    clean_data = transform_sales(raw_data)
    validate_sales(clean_data)
    load_sales(clean_data, config)

Lecture attendue

Le code raconte l’histoire du traitement. Même sans lire les détails, on comprend les étapes.

II.d — Point d’entrée

Un module ne doit pas exécuter un traitement lourd à l’import

Bon point d’entrée

def main():
    config = load_config()
    run(config)

if __name__ == "__main__":
    main()

Pourquoi c’est utile

  • le fichier peut être lancé directement ;
  • les fonctions peuvent être importées dans des tests ;
  • l’orchestrateur peut appeler main() ;
  • l’import charge des définitions, pas le job.
Mauvais réflexe : lancer extract(), transform(), load() au niveau du fichier.
III.a — Sources data

Les formats changent, mais les questions reviennent toujours

Sources classiques

  • CSV, JSON, Excel, Parquet ;
  • APIs ;
  • bases SQL ;
  • exports applicatifs ;
  • stockage objet ;
  • messages ou événements.

Questions de format

  • la donnée est-elle complète ?
  • le format est-il stable ?
  • les types sont-ils cohérents ?
  • que faire si la source est vide ?

Questions d’exploitation

  • le volume est-il raisonnable ?
  • la source peut-elle changer ?
  • quelle sortie attend-on ?
  • quelle anomalie doit bloquer ?
III.b — Ne jamais supposer la donnée propre

Un traitement professionnel anticipe les imperfections

Problèmes fréquents

  • colonne attendue absente ;
  • date au mauvais format ;
  • nombre sous forme de texte ;
  • fichier vide ;
  • doublons ;
  • API qui renvoie une erreur ;
  • champ optionnel ;
  • encodage incorrect.

Fragile vs explicite

# fragile
amount = float(row["amount"])

# plus explicite
def parse_amount(value: str) -> float:
    if value is None or value == "":
        raise ValueError("Missing amount")

    try:
        return float(value)
    except ValueError as exc:
        raise ValueError(f"Invalid amount: {value}") from exc
III.c — Représenter les données

Choisir la structure selon volume, lisibilité et besoin de validation

Liste de dictionnaires

[
  {"date": "2026-06-01", "store": "Paris", "amount": 120.5},
  {"date": "2026-06-01", "store": "Lyon", "amount": 98.0},
]

Simple, lisible, natif Python.

DataFrame pandas

df = pd.DataFrame(records)

Pratique pour manipuler des données tabulaires, mais peut masquer certains problèmes de types.

Objet typé

@dataclass
class Sale:
    sale_date: date
    store: str
    amount: float

Explicite, utile pour clarifier les règles.

Il n’y a pas une seule bonne représentation : le choix dépend du contexte.
III.d — Sortie exploitable

Un traitement ne doit pas “produire quelque chose quelque part”

Sorties possibles

  • fichier CSV nettoyé ;
  • table SQL ;
  • fichier Parquet ;
  • rapport d’anomalies ;
  • statut d’exécution ;
  • métriques de traitement ;
  • donnée prête pour un autre système.

Sortie bien définie

Le job écrit sales_clean_YYYY-MM-DD.csv dans le dossier configuré, avec les colonnes attendues, et produit un rapport d’erreurs si des lignes sont rejetées.

Sortie mal définie : “le script génère un fichier quelque part”.
IV.a — Fonctions pures

Isoler la logique métier des effets de bord

Fonction pure

Dépend uniquement de ses entrées et retourne un résultat. Elle ne lit pas de fichier, n’appelle pas d’API et n’écrit pas en base.

def normalize_store_name(name: str) -> str:
    return name.strip().upper()

Fonction à effet de bord

Interagit avec l’extérieur : fichier, API, base, réseau.

def save_file(df, path):
    df.to_csv(path, index=False)
Mettre le maximum de logique métier dans des fonctions pures, et isoler les effets de bord.
IV.b — Fonctions fourre-tout

Une fonction qui fait tout est difficile à tester et à maintenir

Mauvais découpage

def process():
    # lire fichier
    # nettoyer colonnes
    # convertir dates
    # filtrer lignes
    # écrire fichier
    # envoyer notification
  • test difficile ;
  • debug difficile ;
  • responsabilités mélangées.

Meilleure version

def read_input(path): ...
def clean_columns(df): ...
def normalize_dates(df): ...
def filter_invalid_rows(df): ...
def write_output(df, path): ...

Chaque fonction peut être comprise, testée et remplacée séparément.

IV.c — Nommage

Le nom d’une fonction doit exprimer son intention

Exemples faibles

  • process()
  • clean()
  • do_stuff()
  • format_data()
  • final_data()

Exemples meilleurs

  • normalize_column_names()
  • parse_sale_dates()
  • remove_duplicate_orders()
  • compute_daily_revenue()
  • validate_required_columns()
IV.d — Cas limites

Le code doit décider explicitement quoi faire

Cas à prévoir

  • donnée vide ;
  • colonne absente ;
  • date invalide ;
  • montant négatif ;
  • doublon ;
  • valeur nulle ;
  • devise inconnue ;
  • format inattendu.

Exemple

def compute_average_amount(amounts: list[float]) -> float:
    if not amounts:
        raise ValueError("Cannot compute average on empty list")

    return sum(amounts) / len(amounts)

Retourner 0 sans justification masque une anomalie.

V.a — Hardcode

Les valeurs de contexte ne doivent pas être dispersées dans le code

Valeurs à configurer

  • chemin de fichier ;
  • URL d’API ;
  • nom de base ou table ;
  • seuil métier ;
  • date de traitement ;
  • clé API, identifiant, mot de passe.

Mauvais exemple

API_URL = "https://api.company.com/prod/sales"
OUTPUT_PATH = "C:/Users/admin/Desktop/result.csv"
MIN_AMOUNT = 100
  • peu réutilisable ;
  • risque sécurité ;
  • tests plus pénibles ;
  • modification du code à chaque environnement.
V.b — Types de configuration

Distinguer configuration technique et configuration métier

Technique

  • URL, chemin, host, port ;
  • nom de base ;
  • timeout ;
  • credentials ;
  • répertoire de sortie.

Métier

  • pays à traiter ;
  • seuil d’anomalie ;
  • colonnes attendues ;
  • période ;
  • règle de filtrage ;
  • devise ou type de client.
@dataclass
class TechnicalConfig:
    api_url: str
    output_path: str
    timeout_seconds: int

@dataclass
class BusinessConfig:
    min_amount: float
    expected_currency: str
    required_columns: list[str]
V.c — Variables d’environnement

Centraliser la lecture de configuration dans un module dédié

from dataclasses import dataclass
import os

@dataclass
class Config:
    api_url: str
    output_path: str
    timeout_seconds: int

def load_config() -> Config:
    return Config(
        api_url=get_required_env("API_URL"),
        output_path=get_required_env("OUTPUT_PATH"),
        timeout_seconds=int(os.getenv("TIMEOUT_SECONDS", "30")),
    )

def get_required_env(name: str) -> str:
    value = os.getenv(name)
    if not value:
        raise RuntimeError(f"Missing required environment variable: {name}")
    return value

.env.example

API_URL=https://api.example.com/sales
OUTPUT_PATH=./output/sales.csv
TIMEOUT_SECONDS=30

.env.example est versionné ; .env ne doit généralement pas être versionné.

V.d — Validation de configuration

Échouer tôt vaut mieux qu’un comportement ambigu

Mauvaise situation

Le job tourne pendant 15 minutes puis plante parce que OUTPUT_PATH est absent.

Bonne situation

Le job vérifie au démarrage que les paramètres essentiels existent et sont cohérents.

Variables

Variable obligatoire absente, URL vide, chemin inaccessible.

Types

Timeout non numérique, seuil négatif, valeur non reconnue.

Métier

Configuration incompatible avec le traitement attendu.

VI.a — Typologie des erreurs

Bug, erreur technique et erreur métier ne demandent pas la même réaction

Bug

Le code est faux : mauvaise variable, mauvais type, erreur de logique. Il faut corriger le code.

Erreur technique

Système externe indisponible : API, fichier, base, timeout. Parfois retentable.

Erreur métier

Le traitement tourne, mais la donnée ne respecte pas les règles : fichier vide, colonnes absentes, montant négatif.

Exercice oral : classer une liste d’erreurs, puis décider si on arrête, ignore, rejette, log ou remonte.
VI.b — Exceptions explicites

Nommer les erreurs rend le traitement plus lisible

class DataJobError(Exception):
    pass

class InvalidSchemaError(DataJobError):
    pass

class DataQualityError(DataJobError):
    pass

class LoadError(DataJobError):
    pass
def validate_required_columns(df, required_columns):
    missing = [col for col in required_columns if col not in df.columns]

    if missing:
        raise InvalidSchemaError(
            f"Missing required columns: {missing}"
        )
Une erreur nommée est beaucoup plus claire qu’un simple KeyError perdu dans les logs.
VI.c — Ne pas masquer

Une erreur cachée rend le job dangereux pour l’orchestration

À éviter

try:
    load_data(df)
except Exception:
    pass
try:
    load_data(df)
except Exception:
    print("error")

Meilleure approche

try:
    load_data(df)
except Exception as exc:
    logger.exception("Failed to load data")
    raise LoadError("Data loading failed") from exc

raise ... from exc garde la cause initiale et ajoute un message métier.

VI.d — Comportement en cas d’échec

Chaque erreur importante doit avoir une décision associée

Exemples de décisions

  • fichier absent : arrêter ;
  • fichier vide : erreur métier ou statut no data ;
  • ligne invalide : rejeter ou bloquer selon le besoin ;
  • API indisponible : erreur technique ;
  • table cible inaccessible : arrêter ;
  • taux d’erreurs > 5 % : bloquer.

Question à poser

Le job doit-il continuer, rejeter, alerter ou échouer ?

Cette décision doit être explicite dans le code, pas laissée au hasard.

VII.a — Logging

Remplacer les print() par des logs exploitables

import logging

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s %(levelname)s %(name)s - %(message)s"
)

logger = logging.getLogger(__name__)
logger.info("Starting sales job")
logger.info("Loaded %s rows from source", len(df))
logger.warning("Rejected %s invalid rows", rejected_count)
logger.error("Failed to load output file")
En exécution automatisée, un log doit être horodaté, filtrable et utile au diagnostic.
VII.b — Niveaux de logs

Chaque niveau porte une intention

DEBUG

Détail technique utile en développement.

INFO

Étape normale du traitement : extraction démarrée, lignes reçues.

WARNING

Anomalie non bloquante : lignes rejetées, valeur absente tolérée.

ERROR

Erreur empêchant une partie du traitement : chargement impossible.

CRITICAL existe, mais reste rarement nécessaire dans des petits jobs data.
VII.c — Logs utiles

Un bon log aide à comprendre ce qui s’est passé

Informations à logger

  • source traitée ;
  • période ;
  • lignes lues, transformées, rejetées, chargées ;
  • durée ;
  • configuration principale ;
  • point d’échec.
logger.info(
    "Transformation completed: input_rows=%s output_rows=%s rejected_rows=%s",
    input_count,
    output_count,
    rejected_count
)
VII.d — Logs dangereux

Un log doit aider à diagnostiquer, pas exposer ou polluer

Ne pas logger

  • mots de passe ;
  • tokens ;
  • clés API ;
  • données personnelles ;
  • fichiers entiers ;
  • payloads énormes ;
  • bruit verbeux en production.

Exemple

# à éviter
logger.info("Using token: %s", api_token)

# préférable
logger.info("API token loaded: %s", bool(api_token))
VIII.a — Tests hors orchestrateur

Les tests Python vérifient la logique ; l’orchestrateur vérifie l’exécution du workflow

Pourquoi isoler

  • tests rapides ;
  • diagnostic plus clair ;
  • pas besoin de lancer Airflow ;
  • fonctions vérifiées avant orchestration ;
  • détection précoce des régressions.

À éviter

  • tester toute la logique uniquement dans l’orchestrateur ;
  • rendre chaque test lent et fragile ;
  • mélanger erreur de workflow et erreur métier ;
  • attendre la production pour découvrir les cas limites.
VIII.b — Tester les transformations

Une entrée contrôlée doit produire une sortie attendue

def normalize_amount(value):
    if value is None:
        raise ValueError("Amount is missing")
    return float(value)

def test_normalize_amount_from_string():
    assert normalize_amount("12.5") == 12.5
def remove_negative_amounts(df):
    return df[df["amount"] >= 0]

def test_remove_negative_amounts():
    df = pd.DataFrame({"amount": [10, -5, 20]})
    result = remove_negative_amounts(df)
    assert len(result) == 2
    assert result["amount"].tolist() == [10, 20]
VIII.c — Cas limites

Tester aussi ce qui ne doit pas passer

Cas à tester

  • liste vide ;
  • DataFrame vide ;
  • colonne manquante ;
  • valeur nulle ;
  • doublon ;
  • type incorrect ;
  • date invalide ;
  • montant négatif.
import pytest

def test_normalize_amount_raises_on_missing_value():
    with pytest.raises(ValueError):
        normalize_amount(None)
VIII.d — Erreurs attendues

Un test doit vérifier que le code échoue correctement

def validate_required_columns(df, required_columns):
    missing = [col for col in required_columns if col not in df.columns]
    if missing:
        raise InvalidSchemaError(f"Missing required columns: {missing}")
def test_validate_required_columns_raises_when_column_missing():
    df = pd.DataFrame({"amount": [10]})

    with pytest.raises(InvalidSchemaError):
        validate_required_columns(df, ["amount", "date"])
Un échec prévisible et bien nommé est préférable à un crash inattendu.
IX.a — Traitement appelable

Un job doit pouvoir être lancé par une commande claire

python -m data_job.main --date 2026-06-01 --source sales

Intérêt

  • appelable en local ;
  • paramétrable par un orchestrateur ;
  • contrat d’exécution explicite ;
  • commande documentable dans le README.
import argparse

def parse_args():
    parser = argparse.ArgumentParser()
    parser.add_argument("--date", required=True)
    parser.add_argument("--source", required=True)
    return parser.parse_args()

def main():
    args = parse_args()
    config = load_config()
    run(config=config, run_date=args.date, source=args.source)
IX.b — Statut clair

L’orchestrateur doit savoir si le traitement a réussi ou échoué

Règle simple

  • programme terminé normalement = succès ;
  • exception non rattrapée = échec ;
  • erreur cachée = faux succès dangereux.
def main():
    try:
        config = load_config()
        run(config)
    except DataJobError:
        logger.exception("Data job failed")
        raise
On log l’erreur, puis on la remonte. L’orchestrateur verra l’échec.
IX.c — Indépendance locale

Le job ne doit pas dépendre du poste du développeur

À éviter

  • chemins absolus ;
  • fichiers sur le bureau ;
  • dossier temporaire non maîtrisé ;
  • configuration non documentée ;
  • fichier source supposé présent sans contrôle.

Exemple

# mauvais
path = "C:/Users/Paul/Desktop/input.csv"

# meilleur
path = config.input_path

INPUT_PATH doit être documenté dans .env.example.

IX.d — Exécution répétée

Un job relancé ne doit pas créer un état incohérent

Comportements dangereux

  • ajouter les mêmes lignes à chaque relance ;
  • écraser un fichier sans prévenir ;
  • mélanger les résultats de deux dates ;
  • écrire un fichier partiel comme complet ;
  • confondre fichier temporaire et final.

Bonnes pratiques simples

  • écrire d’abord dans un fichier temporaire ;
  • renommer en final à la fin ;
  • inclure la date de traitement ;
  • contrôler l’existence de la sortie ;
  • logger ce qui a été écrit.
X.a — Versionner

Un job data professionnel doit vivre dans Git

À éviter

  • code envoyé par mail ;
  • script stocké uniquement sur un poste ;
  • script_final_v3_ok.py ;
  • modifications non tracées.

À mettre en place

  • commits lisibles ;
  • branches ;
  • historique ;
  • revue de code ;
  • détection des erreurs avant production.
X.b — README

Documenter le lancement est une partie du travail

Contenu minimal

  • objectif du job ;
  • source lue ;
  • sortie produite ;
  • variables d’environnement ;
  • installation ;
  • commande d’exécution ;
  • commande de test ;
  • erreurs fréquentes.

X.c — Vérification automatique

Préparer progressivement l’intégration continue

Outils possibles

  • pytest pour les tests ;
  • ruff pour le linting ;
  • black pour le formatage ;
  • GitHub Actions ou GitLab CI pour automatiser.

Pipeline mental

  • à chaque modification, lancer les tests ;
  • si les tests échouent, ne pas déployer ;
  • si le code est mal formaté, corriger avant merge ;
  • vérifier avant exécution automatique.
X.d — Passage vers l’orchestrateur

Contrat attendu avant d’être appelé par Airflow, n8n ou autre

Le job doit fournir

  • une commande de lancement claire ;
  • des paramètres explicites ;
  • des logs exploitables ;
  • une exception en cas d’échec ;
  • une configuration documentée ;
  • un code testé ;
  • une sortie connue.

Phrase clé

Si votre traitement ne peut pas tourner proprement sans orchestrateur, il ne tournera pas proprement avec orchestrateur.

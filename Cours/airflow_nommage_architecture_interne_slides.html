1 — Nommage

Pourquoi le nommage est important dans Airflow

Un DAG doit pouvoir être

  • lu rapidement ;
  • compris sans ouvrir tout le code ;
  • supervisé dans l’interface ;
  • diagnostiqué en cas d’échec ;
  • maintenu dans le temps ;
  • expliqué à une autre personne.

Dans un vrai projet

Un mauvais nommage devient vite un problème : on ne sait pas ce que fait chaque tâche, quelle étape est critique, où se situe l’erreur, ni quelle partie peut être relancée.

Dans Airflow, un bon nom permet de comprendre le workflow avant même de lire le code.
1 — Nommage

Un graphe peut fonctionner techniquement et rester illisible

Mauvais DAG

task1 >> task2 >> task3

Ce graphe ne dit rien : aucune intention, aucune donnée, aucun rôle.

DAG plus lisible

fetch_weather_data
>> validate_weather_payload
>> transform_weather_records
>> load_weather_to_postgres
>> write_ingestion_log

Même sans lire le code, on comprend la logique générale.

2 — Nommer un DAG

Le nom du DAG doit décrire le processus orchestré

À éviter

  • test
  • dag1
  • python_dag
  • weather
  • new_pipeline

Bon nom : information minimale

  • le domaine ;
  • le processus ;
  • éventuellement la fréquence ;
  • éventuellement la cible.

Exemples corrects

  • weather_daily_ingestion
  • openmeteo_multi_city_ingestion
  • exchange_rates_daily_pipeline
  • customer_orders_hourly_sync

Fil rouge Open-Meteo

openmeteo_daily_city_weather_ingestion

Source : Open-Meteo. Fréquence : daily. Périmètre : city weather. Type : ingestion.

2 — Nommer un DAG

Précis, mais pas illisible

Trop court

weather

On ne connaît ni la source, ni la fréquence, ni le processus.

Trop long

daily_pipeline_that_fetches_weather_from_openmeteo_for_multiple_cities_and_loads_it_to_postgres

Le nom devient difficile à lire dans l’UI.

Bon équilibre

openmeteo_daily_weather_ingestion

Le nom est spécifique, stable et lisible.

3 — Nommer les tâches

Une tâche commence par un verbe d’action

Verbes utiles

fetchextractvalidatetransformloadcheckwritenotifyarchivegeneratesend

Exemples

  • fetch_weather_data
  • validate_weather_payload
  • transform_weather_records
  • load_weather_to_postgres
  • notify_ingestion_failure

À éviter

weatherdataprocessstep1finalfunction
3 — Nommer les tâches

Le nom doit indiquer l’objet manipulé et rester stable

Vague

validate_data

Acceptable dans un petit TP, mais trop générique dans un vrai pipeline.

Plus clair

validate_weather_payload
validate_openmeteo_response

On comprend immédiatement ce qui est validé.

Nom fragile

call_requests_library

Le nom décrit une implémentation. Si on remplace requests par httpx, il devient faux.

Nom stable

fetch_weather_data

Le nom décrit le rôle dans le pipeline, pas la ligne de code utilisée.

Le nom d’une tâche doit décrire son rôle dans le pipeline, pas uniquement son implémentation.

5 — Règles concrètes

Formules de nommage et checklist

Formule 1

verbe + objet + contexte optionnel

  • fetch_weather_data
  • validate_weather_response
  • load_weather_to_postgres
  • notify_api_failure

Formule 2

action + source/cible + donnée

  • extract_openmeteo_data
  • load_postgres_weather_records
  • generate_weather_quality_report

Checklist

  • Que fait la tâche ?
  • Sur quelle donnée agit-elle ?
  • Est-ce une extraction, une transformation, un chargement ou un contrôle ?
  • Où se situe-t-elle dans le pipeline ?
  • Le nom restera-t-il valable si l’implémentation change légèrement ?
6 — Granularité

Étapes métier vs étapes techniques

Airflow sert à orchestrer des étapes significatives, pas à afficher chaque ligne de code dans le graphe.

Étape métier / pipeline

Action compréhensible dans le processus global, avec une intention fonctionnelle ou opérationnelle.

  • récupérer les données météo ;
  • vérifier la complétude des données ;
  • transformer les données pour analyse ;
  • charger les données en base ;
  • détecter une anomalie ;
  • envoyer une alerte ;
  • écrire une trace d’ingestion.

Détail technique

Opération nécessaire à l’exécution du code, mais généralement trop fine pour apparaître comme tâche Airflow.

  • ouvrir une connexion HTTP ;
  • convertir une chaîne en date ;
  • renommer une clé JSON ;
  • parser une réponse JSON ;
  • initialiser une variable ;
  • créer une liste vide ;
  • convertir des degrés en flottant.
7 — Décider une tâche

Quand créer une tâche Airflow séparée ?

Supervision

Elle doit être observée séparément.

validate_weather_data

Cause d’échec différente

API indisponible vs base indisponible : deux problèmes différents.

fetch_weather_data
load_weather_to_postgres

Relance indépendante

Si le chargement échoue, on peut vouloir relancer uniquement le chargement.

Frontière logique

extractionvalidationtransformationchargementnotificationarchivage

Stratégie d’échec différente

  • appel API : retry pertinent ;
  • validation qualité : retry souvent inutile ;
  • chargement PostgreSQL : retry parfois pertinent ;
  • erreur de schéma : retry inutile.
8 — Ce qui reste dans le code

Ce qui ne mérite généralement pas une tâche séparée

Micro-transformations

  • rename_temperature_field
  • convert_date_string
  • strip_city_name
  • round_temperature

À garder dans transform_weather_records.

Détails d’implémentation

  • open_database_connection
  • close_database_connection
  • create_python_list
  • parse_json

Importants dans le code, pas forcément dans le graphe.

Opérations trop dépendantes du code

call_function_x

Nommer selon le rôle : fetch_weather_data.

9 — Exemple complet

Mauvais découpage : tâche fourre-tout

Cas métier

Récupérer chaque jour la météo de plusieurs villes via Open-Meteo, contrôler les données et charger PostgreSQL.

Découpage

run_weather_pipeline

Cette tâche fait tout : appel API, validation, transformation, chargement, logs, alertes.

Problèmes

  • impossible de voir quelle étape échoue ;
  • impossible de relancer proprement une seule partie ;
  • logs trop mélangés ;
  • supervision faible ;
  • le DAG n’apporte presque rien par rapport à un script classique.
Ce découpage revient à utiliser Airflow comme un simple lanceur de script.

10 — Parallélisation

Découper par ville seulement si cela apporte de la supervision

Exemple

fetch_weather_paris
fetch_weather_lyon
fetch_weather_marseille
        ↓
validate_weather_payloads
        ↓
transform_weather_records
        ↓
load_weather_to_postgres
        ↓
write_ingestion_log

Intérêt

  • les appels API sont indépendants ;
  • un échec sur une ville peut être identifié ;
  • le graphe montre mieux le traitement multi-villes.

Limite

Pour 3 villes, c’est lisible. Pour 300 villes, ce serait peut-être trop lourd, sauf mécanisme adapté comme le mapping dynamique.

La granularité dépend du besoin de supervision, de reprise et de lisibilité.

12 — Architecture interne

Airflow est une plateforme, pas seulement une librairie Python

Airflow permet de

  • lire les workflows ;
  • planifier les exécutions ;
  • lancer les tâches ;
  • stocker les états ;
  • afficher les résultats ;
  • permettre la supervision.

Composants principaux

  • fichiers DAG ;
  • scheduler ;
  • executor ;
  • workers ou processus d’exécution ;
  • metadata database ;
  • webserver ;
  • interface utilisateur ;
  • logs.
13 — Vue d’ensemble

Les composants Airflow travaillent ensemble

Le scheduler planifie, l’executor lance ou délègue, les workers exécutent, la metadata DB mémorise, l’UI expose.
14 — Les fichiers DAG

Les fichiers DAG décrivent les workflows

Ils contiennent généralement

  • la définition du DAG ;
  • les paramètres du DAG ;
  • les tâches ;
  • les dépendances ;
  • la planification ;
  • parfois les appels vers du code métier.

Structure plus propre

dags/
  openmeteo_daily_weather_ingestion.py
include/
  weather_api.py
  weather_transform.py
  postgres_loader.py

Le fichier DAG décrit l’orchestration. La logique métier peut être placée dans des modules séparés.

14 — Exemple DAG

Exemple simplifié de fichier DAG

from airflow import DAG
from airflow.operators.python import PythonOperator
from datetime import datetime

def fetch_weather():
    print("Fetching weather data")

def load_weather():
    print("Loading weather data")

with DAG(
    dag_id="openmeteo_daily_weather_ingestion",
    start_date=datetime(2026, 1, 1),
    schedule="@daily",
    catchup=False,
) as dag:
    fetch_task = PythonOperator(
        task_id="fetch_weather_data",
        python_callable=fetch_weather,
    )

    load_task = PythonOperator(
        task_id="load_weather_to_postgres",
        python_callable=load_weather,
    )

    fetch_task >> load_task
Le fichier DAG ne devrait pas devenir un énorme fichier contenant toute la logique métier.
15 — Scheduler

Le scheduler décide quoi lancer et quand

Rôle

  • quels DAGs doivent être exécutés ;
  • quand ils doivent être exécutés ;
  • quelles tâches sont prêtes ;
  • quelles dépendances sont satisfaites ;
  • quelles task instances envoyer à l’executor.

Exemple

Un DAG est planifié tous les jours à 8h.

  • le DAG est-il actif ?
  • le planning indique-t-il qu’un nouveau run doit être créé ?
  • les dépendances sont-elles respectées ?
  • quelles tâches peuvent démarrer ?
  • quelles tâches doivent attendre ?
Le scheduler est le chef d’orchestre : il décide quoi lancer et quand, mais il ne joue pas lui-même tous les instruments.
16 — Executor

L’executor détermine comment les tâches sont exécutées

Idée centrale

Le scheduler décide qu’une tâche doit être lancée. L’executor se charge de son exécution concrète, directement ou en délégant.

Modes possibles

  • locale ;
  • parallèle sur la même machine ;
  • distribuée sur plusieurs workers ;
  • exécutée dans des conteneurs ou environnements séparés.

SequentialExecutor

Simple, pédagogique, exécution séquentielle.

LocalExecutor

Parallélisme sur la même machine.

CeleryExecutor

Workers distribués.

KubernetesExecutor

Exécution plus industrielle en pods.

Le scheduler planifie. L’executor exécute ou délègue l’exécution.
17 — Workers

Les workers réalisent concrètement le travail

Configuration simple

Le worker peut être sur la même machine qu’Airflow.

Configuration avancée

Plusieurs workers peuvent exécuter plusieurs tâches en parallèle.

Scheduler
   ↓
Executor
   ↓
Worker 1 → fetch_weather_data
Worker 2 → transform_weather_records
Worker 3 → load_weather_to_postgres
Les workers exécutent réellement le code demandé par les tâches.
18 — Metadata DB

La metadata database est la mémoire opérationnelle d’Airflow

Elle stocke

  • les DAG Runs ;
  • les Task Instances ;
  • les états des tâches ;
  • les dates d’exécution ;
  • l’historique ;
  • les connexions ;
  • les variables ;
  • les informations utilisées par l’UI.

Exemples d’états

successfailedrunningqueuedskippedupstream_failed

Sans metadata DB, Airflow ne saurait pas ce qui a été lancé, réussi, échoué ou doit être affiché.

Metadata DB Airflow ≠ base métier du pipeline.
18 — Metadata DB

Ne pas confondre mémoire Airflow et données métier

Metadata DB Airflow

  • état des DAGs ;
  • état des tâches ;
  • historique des runs ;
  • variables et connexions ;
  • informations pour l’UI.

Base PostgreSQL métier

  • données météo ;
  • taux de change ;
  • données transformées ;
  • tables analytiques ;
  • données du projet final.
Airflow orchestre le chargement. PostgreSQL stocke les données métier.
19 — Webserver et UI

L’interface sert à piloter et diagnostiquer

L’UI permet de

  • voir les DAGs disponibles ;
  • activer ou désactiver un DAG ;
  • lancer manuellement un DAG ;
  • consulter les DAG Runs ;
  • voir les tâches ;
  • lire les logs ;
  • comprendre les dépendances ;
  • relancer une tâche ;
  • analyser les échecs.

Limite

L’UI ne remplace pas le code. Elle permet de superviser le code orchestré.

L’interface Airflow sert à piloter et diagnostiquer les exécutions, pas à écrire la logique métier.
20 — Logs

Un pipeline sans logs exploitables est difficile à maintenir

Les logs doivent expliquer

  • ce que la tâche a fait ;
  • à quel moment elle a échoué ;
  • quelle erreur a été levée ;
  • quelles données ou paramètres ont été utilisés ;
  • si une étape a été atteinte ou non.

Logs utiles vs inutiles

Starting Open-Meteo API extraction
Cities: Paris, Lyon, Marseille
API response received for Paris
Validation failed: missing temperature for Lyon
start
ok
done
error
Les logs doivent être suffisamment précis pour diagnostiquer un problème.
21 — Cycle d’exécution

Ce qui se passe quand un DAG est exécuté — étapes 1 à 4

1. Fichier DAG présent

dags/openmeteo_daily_weather_ingestion.py

2. Airflow lit le fichier

Il parse le nom du DAG, les tâches, les dépendances, le schedule et les paramètres.

3. Le scheduler crée un DAG Run

Exemple : exécution du 2026-04-27.

4. Airflow crée des Task Instances

Chaque tâche existe pour ce run précis.

Un DAG Run représente une instance d’exécution du DAG dans le temps.
21 — Cycle d’exécution

Ce qui se passe quand un DAG est exécuté — étapes 5 à 8

5. Tâches prêtes envoyées à l’executor

Si une tâche n’a pas de dépendance bloquante, elle peut partir.

6. Workers exécutent

La tâche peut réussir, échouer, être relancée, ignorée ou attendre une ressource.

7. États enregistrés

fetch: success
validate: success
load: failed

8. UI affiche

Graphe, états, logs, erreurs et historique.

22 — Concepts fondamentaux

DAG, DAG Run, Task et Task Instance

DAG

Modèle du workflow : tâches, dépendances, planning.

openmeteo_daily_weather_ingestion

DAG Run

Exécution particulière du DAG dans le temps.

Run du 2026-04-27

Task

Étape définie dans le DAG.

fetch_weather_data

Task Instance

Exécution d’une tâche pour un DAG Run donné.

fetch_weather_data — 2026-04-27

DAG et task décrivent le workflow. DAG Run et task instance décrivent son exécution réelle.

25 — Erreurs fréquentes

Erreurs à corriger chez les étudiants

Confusions sur le rôle d’Airflow

  • Croire qu’Airflow stocke les données métier.
  • Croire qu’Airflow transforme automatiquement les données.

Airflow orchestre le chargement et lance du code, mais ne remplace ni PostgreSQL, ni Python, ni SQL.

Confusions sur les tâches

  • Mettre toute la logique dans une seule tâche.
  • Découper chaque ligne de code en tâche.
  • Confondre tâche Airflow et fonction Python.

La fonction est du code. La tâche est une unité orchestrée, suivie, historisée et supervisée.

26 — Synthèse

Ce qu’est un bon DAG Airflow

Un bon DAG doit être

  • lisible ;
  • correctement nommé ;
  • découpé selon des étapes significatives ;
  • compréhensible depuis l’UI ;
  • capable de montrer clairement où se situe un échec ;
  • assez détaillé pour superviser ;
  • pas trop détaillé pour rester maintenable.

Airflow repose sur

  • DAG files : décrivent les workflows ;
  • Scheduler : décide quand lancer ;
  • Executor : lance ou délègue ;
  • Workers : exécutent les tâches ;
  • Metadata DB : conserve les états ;
  • Webserver : affiche les informations ;
  • Logs : permettent de diagnostiquer.
Airflow sert à rendre un pipeline observable, structuré, relançable et maintenable. Le code exécute le traitement ; Airflow organise son exécution.
